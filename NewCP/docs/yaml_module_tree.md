# YAML-corpus module map: imports, exports, and porting order

## Purpose

This document inventories the 272 Component Pascal modules in the lifted `.odc` corpus
under [`YAML/`](../../YAML), groups them by subsystem, captures the import graph, and
proposes a porting order to NewCP. The data was generated by parsing every
`*.odc.yaml` text run and extracting `MODULE` / `IMPORT` / `PROCEDURE*` / `TYPE*`
declarations. Re-generate with [`tools/extract_imports.py`](../../tools/extract_imports.py),
[`tools/extract_provides.py`](../../tools/extract_provides.py), and
[`tools/analyze_imports.py`](../../tools/analyze_imports.py).

Module-naming convention: with the exception of `System/Mod/*` (whose modules use bare
names — `Kernel`, `Files`, `Stores`, `Views`, ...) every other subsystem prefixes its
modules with the subsystem name. So `Std/Mod/Cmds.odc` declares `MODULE StdCmds;`,
`Text/Mod/Views.odc` declares `MODULE TextViews;`, etc. The doc uses the *declared*
module names everywhere — those are the names other modules import.

## Subsystem-level dependency graph

Edge `A -> B` means at least one module in `A` imports at least one module in `B`.

```
  Com      -> Std, System, Win
  Comm     -> Std, System, Text, Win
  Ctl      -> System, Win
  Dev      -> Form, Host, Std, System, Text, Win
  Form     -> Std, System
  Host     -> Ole, Std, System, Text, Win
  Obx      -> Ctl, Form, Host, Ole, Std, System, Text, Win
  Ole      -> Ctl, Host, Std, System, Text, Win
  Sql      -> Host, System, Text, Win
  Std      -> Form, Host, System, Text
  System   -> Host, Ole, Std, Text, Win
  Text     -> System
  Xhtml    -> Std, System, Text
```

Notes:

- `Text` depends only on `System`. **Text is the first non-System subsystem to port.**
- `Form` depends only on `Std` and `System`. Port after Std.
- `System` shows back-edges to `Text`, `Std`, `Host`, `Ole`, `Win` — these come from a
  small set of *facade* / *tutorial* modules and are not part of the core dep chain:
    - `Out`, `In` (tutorial wrappers; depend on `TextMappers` / `StdLog`)
    - `XYplane` (Wirth-book pixel surface; depends on `HostPorts` / `HostWindows`)
    - `Init` (depends on `HostMenus`)
    - `Config` (depends on `OleData`)
    - `Controls` (depends on `StdCFrames`)
    - `Kernel` (depends on `WinApi`, `WinOle` — Win-specific bindings)
  None of these are required to port the System core; they can be deferred or
  re-stubbed in NewCP.

## Recommended porting tiers

Aligned with the existing roadmap (Phase 5 = "Runtime compatibility shell";
Phase 7 = "Framework recovery"). Each tier is a coherent chunk that can be
delivered together; later tiers cannot start until earlier tiers can compile
and load.

### Tier 0 — Runtime substrate (Rust first, then CP)

Present as Rust services per the roadmap; each can be replaced by a CP module once
the compiler is self-hosting. Layer 0..3 of the graph.

| Module | Why first | Notes |
|---|---|---|
| `Kernel` | The runtime — every other module depends on it transitively | Currently Rust-resident. Replace last among Tier 0. |
| `Math`, `SMath` | Leaf, no deps | Can come from a libm shim. |
| `Files` | Persistence + Stores foundation | Will sit on `HostFiles` (Win32) or a new portable backend. |
| `Log` | Single global text buffer; trivial dep on Kernel | Already exists in `NewCP/Mod/Log.cp` as a placeholder. |
| `Dates` | Used by Stores/Sequencers logging | Pure CP. |
| `Services` | Action queue, timers — used by Views | Needs a host event loop hook. |
| `Strings` | String <-> int/real utility | Pure CP. |
| `Meta` | Reflection over modules/types/procs | Reads module descriptors from `Kernel`. |
| `Dialog` | Notifier protocol; everyone uses it | Modal facade can stub `Beep`/`MsgBox`. |

Everything else: `Comm`, `Ole`, `Sql`, `Ctl`, `Com`, `Win` (beyond `WinApi`/`WinOle`),

- `Win*` FFI modules — port any one as soon as a CP module needs that API. `WinApi`

- `Obx*` — examples; convert opportunistically as integration tests.
- `Sql`, `Ole`, `Ctl`, `Com`, `Comm`, `Xhtml` — port only when a target app needs them.

## Methodology and caveats

- The text content of each `.odc.yaml` is reassembled by concatenating every
  `- text:` run in document order. This recovers the source faithfully because
  the YAML reflects the BlackBox text-run model. Inline non-text Views (folds,
  hyperlinks, etc.) are skipped — they are decorations, not source.
- IMPORT clauses are matched as `IMPORT ... ;` non-greedily; both `Name` and
  `Alias := Name` forms are recovered. `(* ... *)` comments inside the IMPORT
  clause are stripped before parsing.
- Exports are counted by regex (`PROCEDURE ... Name*` and `Name* = ...`). The
  counts are approximate but useful for spotting which modules are weighty
  (`Kernel` 89 procs / 17 types; `Views` 70 procs / 14 types).
- A handful of leaf modules really do have no IMPORT clause at all (e.g.
  `Fonts`, `WinGL`, `WinSql`, `WinCmc`, `ObxRandom`). They appear in Layer 0.
- The corpus is BlackBox 1.7. If you target a different release, regenerate
  the YAML (see [`YAML/README.md`](../../YAML/README.md)) before re-running
  the analyzers.
